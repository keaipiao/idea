# 开发流程 SOP

> **本文档是项目元规范**，定义"每个功能从立项到上线"的标准流程，确保任何功能都不再随机化。
>
> 修改本文档前先和 piaopiao 对齐 —— 修订后需同步更新 `~/.claude/skills/piao-workflow/SKILL.md`。
>
> 配套资源：
> - 新功能起手：复制 `docs/_功能模板/` → `docs/0X-功能名/`
> - 冷启动部署：见 `docs/_部署手册.md`
> - 跨功能踩坑：见 `docs/_部署踩坑记录.md`

---

## 一、文档骨架：每个功能的 5 件套

每个 `docs/0X-功能名/` 目录必须且只有这 5 份文档（外加可选的 `资源/` 子目录）：

| 序号 | 文档 | 业内对应 | 解决什么问题 | 何时写 |
|---|---|---|---|---|
| 01 | **设计.md** | PRD + Design Doc | 做什么 / 为什么 / 怎么做 | 阶段 1-4 持续填充 |
| 02 | **开发计划.md** | Implementation Plan | PR 怎么拆 / 谁先谁后 / 回滚边界 | 阶段 5 |
| 03 | **测试.md** | Test Plan + Report | 怎么验证 / 通过情况 | 阶段 6 写计划，阶段 7 填报告 |
| 04 | **部署日志.md** | Runbook（per-release snapshot） | 本功能上线那次的步骤 / 迁移 / 回滚 | 阶段 8-9 |
| 05 | **迭代日志.md** | Changelog（per-feature） | 该功能历次 PR 演化、release notes | 阶段 9 起，每次发版追加 |

**`01-设计.md` 的标准章节**（业内 PRD + Design Doc 合并写法）：
1. 需求（做什么、为什么、不做什么）
2. 调研笔记（行业现状、竞品、技术选型）
3. UI 设计（如涉及前端：线框图引用 + 设计规范）
4. 技术设计（数据模型、接口契约、关键算法、迁移策略、ADR 决策记录）

**`05-迭代日志.md` 写法**：按时间倒序，每次发版一节，标题为 `YYYY-MM-DD · vX.Y.Z · 一句话摘要`。开发过程笔记（PR 完工总结、阶段需求确认等）也合并进来，不开子目录。

## 三、SOP 主流程：10 阶段

本流程和 `piao-workflow` skill 严格对齐。每个新功能从阶段 0 起步，**不许跳阶段、不许并行**。

| # | 阶段 | 必做动作 | 产出 / 更新到 | 退出条件 |
|---|---|---|---|---|
| **0** | **起手** | 复制 `docs/_功能模板/` → `docs/0X-功能名/`（X 取下一个未用序号） | 5 件套空模板就位 | 目录建好 |
| **1** | **需求确认** | 和用户对齐"做什么/为什么/不做什么"；先评价方案再给建议 | `01-设计.md` § 1 需求 | 用户认可需求 |
| **2** | **调研** | `ctx7` 拿技术文档 + `firecrawl` 查行业/竞品；grep 项目内既有相似模式 | `01-设计.md` § 2 调研笔记 | 关键不确定性消除 |
| **3** | **UI 设计**（涉前端必经） | 看已有 UI 规范；无则调 `plan-design-review` 出线框/原型 | `01-设计.md` § 3 UI + `资源/线框图/` | 用户认可 UI |
| **4** | **技术设计** | 数据模型、接口契约、关键算法、迁移、ADR | `01-设计.md` § 4 技术设计 | `plan-eng-review` 通过 |
| **5** | **开发计划** | 拆 PR（最小粒度）、估时、定每个 PR 的回滚边界 | `02-开发计划.md` | 用户认可拆分 |
| **6** | **实现** | 按 PR 顺序写码 + 单元测试 + 每 PR 单独 commit；中文注释；禁魔法值 | 代码 + `05-迭代日志.md` 追加一段 PR 摘要 | 该 PR 自测通过 |
| **7** | **测试** | 单测全绿 + Claude in Chrome 真实浏览器联调金路径和边界 | `03-测试.md` | 全部用例 pass + console 0 error |
| **8** | **部署清单** | 列迁移顺序、env 变更、nginx 改动、回滚预案；本地起 prod stack 预演 | `04-部署日志.md` | 用户审过清单 |
| **9** | **Ship 上线** | 服务器部署 + smoke test + 浏览器 E2E。同步更新 `_部署手册.md`（如有环境变更）+ `_部署踩坑记录.md`（如有新坑）+ `05-迭代日志.md` 写 release note + **根目录 `README.md` + `CHANGELOG.md`**（见规则 9） | 上述 5 份文档同步 | 线上跑通 |
| **10** | **归档 + Skill 自维护** | 本次踩坑/工具/流程改进沉淀回 `piao-workflow` SKILL.md 的对应阶段 | skill 文件更新 | 经验沉淀完成 |

## 四、关键规则（让流程"硬"起来）

1. **🛑 阶段闸门（Stop-and-Wait Gate）**：每个阶段完成判定后 AI 必须**停止输出**，等用户明确说"OK / 通过 / 继续"或给修改意见，**再**进入下一阶段。
   - 不许把多个阶段产出串在同一次回复里
   - 不许把"用户没反对"解读为"用户同意"（沉默 ≠ 同意）
   - 阶段内部连续小步骤可以连贯执行；跨阶段必须停
   - 用户明确说"自己拿主意 / 不用问我 / 一口气做完"时才允许连贯执行
   - 来源：2026-05-19 用户反馈"提了意见之后不会停止，需要我手动打断才能告诉他想法"
2. **强制起手**：新功能必须从复制 `_功能模板/` 开始 —— 不许凭空建空目录。
3. **阶段不可跳**：0→10 顺序不可乱。需求没确认前不准开始调研，调研没做完不准开始设计。允许阶段内迭代，不许跨阶段并行。
4. **每阶段产出 = 退出条件**：没有产出落到对应文档章节，就不算这阶段完成。
5. **功能边界严格**：跨功能依赖时，只在被依赖功能的文档处展开完整规格，调用方仅以"已在 X 文档定义"备注引用，不重复规格化（CLAUDE.md 全局规范第 11 条）。
6. **关联变更一致性**：改任一功能时检查上下游所有代码 / 文档 / 页面，一并修改（全局规范第 16 条）。
7. **踩坑分层**：
   - 单功能内部一次性坑 → 功能内 `04-部署日志.md`
   - 跨功能可复用运维坑 → 顶层 `_部署踩坑记录.md`
8. **`_部署手册.md` 同步规则**：阶段 9 上线时，若引入新环境变量/容器/nginx 配置/数据库迁移 → 必须把"如何从 0 到此版本一遍部署"对应章节累积到 `_部署手册.md`，让它永远停在 HEAD。
9. **根目录 `README.md` + `CHANGELOG.md` 强制同步规则**（2026-05-23 教训）：阶段 9 归档时**必须**同步更新这两个根目录标准文档，否则新人接手时根目录显示的项目状态会严重过时（首版上线 2 周后根目录 README 仍停在脚手架阶段，CHANGELOG 缺 v0.2-v0.6 中间几个大版本）。
   - **CHANGELOG.md**：新增 `## [vX.Y.Z] — YYYY-MM-DD · 功能名 — docs/<NN>-功能名/` 段，从功能 `04-部署日志.md` 提炼 5-15 行简化版（Added 3-5 bullet / Migrations / Pitfalls Logged 链接到 \_部署踩坑记录.md 条目）。详细变更不在 CHANGELOG 重复，指向功能目录即可
   - **README.md**：检查 4 处是否同步——① 项目状态表新增本次功能行 ② 仓库结构如有新目录补上 ③ 快速开始如有新必填 env 加一行 ④ 关键设计决策表加一行
   - **版本号约定**：项目用 `vX.Y.Z` semver-like，大功能上线 = minor bump（v0.4.0 → v0.5.0），feature flag 切换 / 小修 = patch bump（v0.4.0 → v0.4.1）。版本号在 `04-部署日志.md § 1 上线概览` 显式声明，阶段 9 抄到 CHANGELOG

## 六、命名约定

- **功能目录**：`0X-功能名/` —— 2 位序号 + 中文功能名，序号一旦分配不再变
- **5 件套文件名**：固定 5 个（`01-设计.md` / `02-开发计划.md` / `03-测试.md` / `04-部署日志.md` / `05-迭代日志.md`），不许加新文件占用 6+ 序号
- **资源附件**：放 `资源/` 子目录，内部自由组织
- **commit 信息**：中文 + 类型前缀（feat / fix / refactor / test / docs / chore）
