## D1. Druid wall filter 与 PostgreSQL partial unique index DDL 冲突(2026-05-28 · PR-1)

### 现象

Spring Boot 启动时 Flyway migration 失败:

```
sql injection violation, dbType postgresql, druid-version 1.2.24,
syntax error: not supported.pos 68, line 1, column 64, token WHERE :
CREATE UNIQUE INDEX uk_user_union_id ON t_user(union_id) WHERE union_id IS NOT NULL
```

Spring Boot context init failed → mvn spring-boot:run exit code 1 → 应用无法启动。

### 根因

Druid `wall` filter(SQL 防火墙)对 PostgreSQL 的 **partial index DDL**(`CREATE INDEX ... WHERE`)解析不支持,误判为 SQL injection 攻击拦截。Druid 1.2.x 系列均存在(`com.alibaba.druid.wall.WallFilter`)。

实际 PG partial index 是合法 DDL,用于"仅对非 NULL 值建唯一索引"等业务场景。

### 修复

**dev / test profile**:关 wall filter。

```yaml
spring:
  datasource:
    druid:
      filter:
        wall:
          enabled: false
```

**理由**:
- MyBatis-Plus 全参数化查询已免疫普通 SQL 注入
- wall filter 是 paranoid 防御,在 MVP / 个人项目里成本大于收益
- 真要 wall,prod profile 单开 + 自定义 `wall.config.commentAllow=true` / `selectAllColumnAllow=true` 等

### 防御

- 引入 Druid 后第一次启动时**先在最小 schema 上验证**,不要等业务表全建完才启
- 评审 schema migration 时显式问"DDL 是否用了 partial index / 复杂表达式索引"
- 团队规范:dev 期 wall.enabled=false,prod 期慎用且必须配 SQL 白名单

### 关联

- 配置位置:`src/main/resources/application-dev.yml`(spring.datasource.druid.filter.wall)
- 项目首次出现:PR-1 后端脚手架,Flyway V1__create_tables.sql 第 23 行起的 `CREATE UNIQUE INDEX ... WHERE` 子句

## D2. Testcontainers Java 在 Windows + Docker Desktop WSL2 backend 探测失败(2026-05-28 · PR-1)

### 现象

集成测试启动时 Testcontainers 报错:

```
java.lang.IllegalStateException: Could not find a valid Docker environment.
Please see logs and check configuration
  at org.testcontainers.dockerclient.DockerClientProviderStrategy.lambda$getFirstValidStrategy$7
```

但 `docker ps` 工作,Docker Desktop 已启动,`docker context ls` 显示 `desktop-linux *` 当前。

### 根因

Docker Desktop WSL2 backend 在 Windows 上暴露的 named pipe 是 `\\.\pipe\dockerDesktopLinuxEngine`,**不是** Testcontainers 默认探测的 `\\.\pipe\docker_engine`。Testcontainers Java 1.21.3 的 NpipeSocketClientProviderStrategy 不识别 desktop-linux pipe 路径。

切到 `default` docker context(`docker context use default`)也无效 — Testcontainers 的 socket strategy 硬编码默认 pipe 路径。
设置 `DOCKER_HOST=npipe:////./pipe/dockerDesktopLinuxEngine` 也无效。

### 修复(当前 PR-1 采用)

**改用本机 docker-compose 共享 PG + 独立 `ideabox_test` database**:

```java
@DynamicPropertySource
static void overrideProps(DynamicPropertyRegistry r) {
    r.add("spring.datasource.url", () -> "jdbc:postgresql://localhost:5432/ideabox_test");
    ...
}
```

测试类加 `@org.springframework.transaction.annotation.Transactional`,Spring Boot Test 默认 rollback。

**预置步骤**:
```bash
docker exec ideabox-postgres-dev psql -U ideabox -d postgres -c "CREATE DATABASE ideabox_test OWNER ideabox;"
```

### 影响 / 已知 limitations

- 测试 + dev 共享同一 PG 实例,sequence 推进(`t_user_id_seq` 等)但不影响业务
- 不同 dev 机环境差异:CI / Linux 上 Testcontainers 应该 work,可加 profile gate 兼容

### 防御

- Java 项目用 Testcontainers 前,先在目标平台试探一次空 container.start() 确认环境
- 文档明确"Windows Docker Desktop WSL2 backend 不兼容 Testcontainers"
- 替代:GitHub Actions Linux runner 上无此问题,可作为 CI 集成测试主战场

### 关联

- 配置位置:`src/test/java/com/ideabox/api/IntegrationTestBase.java`
- 项目首次出现:PR-1 后端脚手架 阶段 4 集成测试

## D4. Spring Security CorsConfigurer 优先按 bean name "corsConfigurationSource" 查找(2026-05-28 · PR-3)

### 现象

Spring Security `.cors(Customizer.withDefaults())` 配置后,所有 OPTIONS preflight 返 403 "Invalid CORS request",即便 `allowedOriginPatterns: ["*"]` 也拒绝。
单测 MockMvc 不暴露此问题(MockMvc 不走真 CORS filter chain),只有真浏览器 e2e 联调能逮。

### 根因

`org.springframework.security.config.annotation.web.configurers.CorsConfigurer.getCorsConfigurationSource(H http)` 源码逻辑:
1. 优先按 bean name `"corsConfigurationSource"` 查 ApplicationContext
2. fallback:`getBeansOfType(CorsConfigurationSource.class)` 找到 1 个就用

我们用 `@Bean @Profile("dev") public CorsConfigurationSource devCorsConfigurationSource()`,bean name 是方法名 `devCorsConfigurationSource`。理论 fallback 路径应生效,但实际 fallback 在某些 Spring 6.x 版本上不可靠。

### 修复

显式指定 bean name:
```java
@Bean(name = "corsConfigurationSource")
@Profile("dev")
public CorsConfigurationSource devCorsConfigurationSource() { ... }
```

### 防御

- bean 类型在 Spring 框架有"惯例名"时,**显式 `@Bean(name = "...")`**,不依赖方法名
- 真浏览器 e2e 联调是 BLOCKING — MockMvc 单测不能替代
- 增 CORS 集成测试用 TestRestTemplate(走真 servlet filter chain),覆盖 cross-origin OPTIONS

### 关联

- 配置位置:`src/main/java/com/ideabox/api/config/CorsConfig.java`
- 项目首次出现:PR-3 网页端联调,Next.js 跑 3001 端口时所有 OPTIONS 403

## D5. framer-motion `layout` prop 与 @dnd-kit transform 冲突(2026-05-28 · PR-3)

### 现象

IdeaItem 用 `motion.div` + `layout` prop 包裹 + `useSortable` transform。拖拽后卡片"先回到原位,300ms 后才跳到新位置"。

### 根因

framer-motion 的 `layout` prop 用自己的 FLIP 动画(`getBoundingClientRect` + transform),会**覆盖**外层 `style.transform` 和 `style.transition`(dnd-kit 用的就是这两个)。两者职责互斥:dnd-kit 内部已用 transform 做 layout(`defaultAnimateLayoutChanges`),再叠加 framer-motion `layout` 直接抢资源。

### 修复

移除 `motion.div` + `layout` prop,改回 `<div>`,让 dnd-kit `defaultAnimateLayoutChanges` 处理。framer-motion 仅用在**独立**组件(Toast/Modal/EmptyState/SVG)。

### 防御

- dnd-kit 文档明确"animateLayoutChanges 自带 layout 动画",不需额外 framer-motion `layout`
- 任何"拖动 + layout 动画"组合,**首选 dnd-kit 内置**,framer-motion 仅做独立组件的进入 / 退出动画

### 关联

- 实现位置:`web/app/_components/IdeaItem.tsx` / `web/app/_components/SortableIdeaList.tsx`
- 项目首次出现:PR-3 阶段 5 联调

## D6. 乐观更新中 debounce 把 UI 更新和 API 调用一起 debounce → 用户看到"回弹再跳"(2026-05-28 · PR-3)

### 现象

用户拖完卡片,UI 卡片闪回原位,300ms 后才跳到新位置(因为我们 debounce 了 300ms 调 reorder API,且 SWR cache 也在那时才乐观更新)。

### 根因

`scheduleReorder(nextIds)` 内部 setTimeout 300ms 才调 `onReorder(ids)`,`onReorder` 同时包含:
- `applyOptimisticReorder(allIds)` — 更新 SWR cache
- `reorderApi(allIds)` — 调后端

两个都被 debounce 了。但乐观更新本质应**立即**触发让用户看到"瞬间换位"。

### 修复

拆两个 callback:
- `onLocalReorder(ids)` — **立即**调,更新本地 SWR cache(乐观)
- `onApiReorder(ids)` — debounced 300ms 调后端

```typescript
const handleDragEnd = (e) => {
  const next = arrayMove(ids, oldIndex, newIndex);
  onLocalReorder(next);   // 立即
  scheduleApi(next);       // debounced
};
```

### 防御

- **乐观更新永不 debounce**;debounce 仅针对网络请求节流
- 任何"用户操作 + 后续 API"模式都应思考"哪些立即生效,哪些 debounce"

### 关联

- 实现位置:`web/app/_components/SortableIdeaList.tsx` / `web/app/_components/IdeaList.tsx`
- 项目首次出现:PR-3 阶段 5 用户反馈

## D3. MyBatis-Plus 默认 updateStrategy=NOT_NULL 导致 set null 静默失效(2026-05-28 · PR-1)

### 现象

业务逻辑"取消完成"(把 `completed_at` 字段从有值 set 回 null)未生效:数据库里字段仍然保留旧值,前端列表过滤 `?completed=false` 拿不到这条记录。

但 Service 内 `entity.setCompletedAt(null)` + 返回值都看似正确(因为 in-memory entity 已 null)— 难以察觉。

### 根因

MyBatis-Plus 全局 `updateStrategy=NOT_NULL`(默认),`updateById(entity)` 时**跳过所有 null 字段**,SQL 里不包含 `SET completed_at = NULL` 子句,DB 残留旧值。

### 修复

字段级显式覆盖策略:

```java
@TableField(updateStrategy = FieldStrategy.IGNORED)
private LocalDateTime completedAt;
```

`IGNORED` 表示无视全局策略,无论字段是否 null 都 update。

### 防御

- entity 设计阶段:任何"可以从有值变回 null"的语义字段(取消完成、撤销标签、清除关联),必须 `@TableField(updateStrategy = IGNORED)` 显式声明
- code review:扫描所有 entity 字段,看是否有"可能被 set null"的语义,标记
- 集成测试:覆盖完成→取消完成往返(我们的 `IdeaIntegrationTest.ideaGoldenPath` step 3-5 已覆盖)

### 关联

- 实体定义:`src/main/java/com/ideabox/api/idea/entity/Idea.java`(`completedAt` 字段)
- 项目首次出现:PR-1 IdeaService.update,集成测试发现
